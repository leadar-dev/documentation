# error handling conventions

общие принципы обработки ошибок для всех сервисов leadar.  
языковая специфика — в `python/ERRORS.md`, `rust/ERRORS.md`, `js/ERRORS.md`.

---

## принципы

### не глотаем ошибки
каждая ошибка либо обрабатывается, либо пробрасывается наверх.  
пустой catch/unwrap-и-ignore — запрещены.

### fail fast при старте
если сервис не может стартовать (нет БД, нет брокера, невалидный конфиг) — падаем сразу с понятным сообщением. не пытаемся работать в деградированном режиме.

### логируем один раз
логируем там где **принимаем решение** что делать с ошибкой — retry, skip, остановка.  
не логируем при каждом пробросе вверх по стеку — это создаёт дубли.

### контекст в ошибке
ошибка должна объяснять **что** произошло и **где**. голое "error occurred" бесполезно.

```
❌ "parse error"
✅ "failed to extract stateData from kwork listing page 3"
```

### один элемент не роняет весь цикл
если упал один заказ из листинга — логируем, пропускаем, идём дальше.  
если упала инфраструктура (БД, брокер) — останавливаемся.

## классификация ошибок

| тип | примеры | поведение |
|---|---|---|
| **доменная** | заказ не найден, невалидный payload | обрабатываем, возвращаем клиенту |
| **инфраструктурная** | БД недоступна, брокер упал | retry или стоп |
| **парсинга** | маркер не найден в HTML, неожиданная структура | skip элемента + warning |
| **конфигурации** | не задана переменная окружения | fail fast при старте |

---

## уровни логирования ошибок

| ситуация | уровень |
|---|---|
| пропускаем один элемент, продолжаем | `warning` |
| операция упала, сервис продолжает работу | `error` |
| сервис не может продолжать работу | `critical` |

---

## retry — когда и сколько

| ситуация | retry |
|---|---|
| HTTP 429 (rate limit) | да, с exponential backoff |
| HTTP 5xx (server error) | да, 3 попытки |
| HTTP 4xx (client error) | нет |
| БД временно недоступна | да, с backoff |
| невалидный JSON/HTML | нет — это не временная проблема |
| публикация в брокер упала | да, 3 попытки |

## что пользователь видит

внутренние детали инфраструктурных ошибок **не светим клиенту**:

```
❌ "sqlx error: connection refused to postgres:5432"
✅ "internal server error"

❌ "serde_json parse error at line 42"
✅ "invalid request"

✅ "want 3148262 not found"  — доменная ошибка, можно показать
```
